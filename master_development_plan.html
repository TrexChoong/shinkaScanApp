Choosing the next task

Solo project, so the WIP limit is one: finish what's in progress before starting anything new. When you need a task, run this filter chain top to bottom — stop at the first rule that leaves a single issue.

  1. Only unblocked work is eligible. Open, and every "Blocked by" issue in its GitHub sidebar closed. The Up next list below computes this from the dependency graph and live open/closed state.
  2. Spikes before build work. #6 gates every M1 issue; #25 gates all of M2 and shapes M3. A day of building while a spike waits is a day bet on an undecided architecture.
  3. Current milestone first. Don't reach forward unless the open milestone has nothing eligible — the one designed exception is #25, whose window intentionally opens during M1's buffer week.
  4. Highest priority: P0 → P1 → P2.
  5. Biggest unblocker. Prefer the issue that releases the most downstream work (closing #9 frees four issues; closing #15 frees none).
  6. Earliest scheduled start, then smallest estimate — stay on schedule, and when equal, bank the quick win.

Within an issue, work sub-issues top to bottom — they're listed in dependency order. If you get hard-blocked mid-task, comment exactly where you stopped, move it back to Backlog, and re-run the chain, preferring the same area:* to keep context warm.

Keeping the plan honest

  • This page is a read-only snapshot. Status is pulled live from GitHub when opened from the repo copy; the board is the source of truth.
  • Structure (dates, dependencies, new issues) is baked at generation time — regenerate the plan when it changes; the status line flags new issues since the snapshot.
  • Slipping? Use the milestone buffer first, then the named pressure valves. Never silently move a due date — re-plan the chain.
